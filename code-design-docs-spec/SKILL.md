# 设计文档规范技能

本技能提供三套文档规范，覆盖软件设计的主要文档类型：

1. **架构设计文档** (`arch-spec.md`) - 系统架构设计、模块边界、技术决策
2. **产品需求文档** (`prd-spec.md`) - 产品需求定义、用户故事、功能规格
3. **测试文档** (`test-spec.md`) - 测试策略、测试用例、测试报告

---

## 何时使用本技能

**REQUIRED** 当用户有以下需求时，使用对应规范：

- **架构设计** → 引用 `arch-spec.md`
  - 新系统或子系统从零设计
  - 模块边界划分、接口定义
  - 技术决策记录
  - 核心实体建模

- **产品需求** → 引用 `prd-spec.md`
  - 新产品或功能规划
  - 用户故事编写
  - 需求范围界定
  - 验收标准定义

- **测试规划** → 引用 `test-spec.md`
  - 测试策略制定
  - 测试用例设计
  - 测试报告撰写
  - 覆盖率目标设定

## 使用流程

### 步骤 1：判断文档类型

根据用户需求，确定需要编写的文档类型：

```
用户意图分析
├── 架构/技术设计 → arch-spec
├── 产品/功能需求 → prd-spec
└── 测试相关 → test-spec
```

### 步骤 2：引用对应规范

读取对应文档规范文件：

- `references/arch-spec.md` - 架构设计文档规范
- `references/prd-spec.md` - 产品需求文档规范
- `references/test-spec.md` - 测试文档规范

### 步骤 3：执行规范要求

按照规范中的 Agent 执行工作流：

1. **判断触发条件** - 确认是否需要编写/更新文档
2. **确定文档结构** - 根据复杂度选择章节
3. **按规范写作** - 遵循格式要求和内容边界
4. **执行自检清单** - 逐项检查确保质量

## 规范速查

### 共同约定

所有三套规范遵循相同的标记规则：

- **REQUIRED** - 必须遵守
- **PROHIBITED** - 明确禁止
- **OPTIONAL** - 可选，视情况使用

### 文档状态

所有设计文档统一使用状态标注：

```markdown
> **Status**: `draft` | `active` | `deprecated`
> **Superseded by**: [链接]（仅 deprecated 文档填写）
```

**状态规则**：
- `draft → active`：自检清单全部通过后标记为 `active`
- `active → deprecated`：指向替代文档；30 天以上无替代文档则直接删除

## 规范关联关系

三套规范相互关联，形成完整的设计文档体系：

```
产品需求 (PRD)
    ↓ 驱动
架构设计 (Arch)
    ↓ 约束
测试规划 (Test)
```

**引用关系**：
- PRD 引用架构文档说明技术可行性
- 测试文档引用 PRD 作为测试依据
- 架构文档引用 PRD 确认需求范围

---

## 自检要求

完成文档后，**REQUIRED** 执行对应规范的自检清单：

- **arch-spec** → Part 6 自检清单（总览/子模块/参考文档检查项）
- **prd-spec** → Part 6 自检清单（内容完整性/质量/风险检查项）
- **test-spec** → Part 6 自检清单（计划/用例/报告/质量检查项）

所有检查项必须通过，未通过项 **REQUIRED** 修正后重新检查。

---

## 常见错误

**PROHIBITED** 以下做法：

- 混淆文档类型（在 PRD 中写技术实现细节）
- 跳过触发条件判断（为不需要文档的变更编写文档）
- 忽略自检清单（未检查直接交付）
- 使用模糊语言（"可能"、"大概"、"视情况而定"）
- 复制其他文档内容（应使用链接引用）

## 示例

### 架构设计场景

用户："我需要设计一个新的支付模块"

执行流程：
1. 识别需求 → 架构设计 → 引用 `arch-spec.md`
2. 检查触发条件（新模块暴露接口）→ 满足条件
3. 判断文档类型 → 子模块文档
4. 按 Part 4.3 结构写作：职责定位 → 核心原则 → 边界与实体 → 关键流程 → 设计决策
5. 执行 Part 6.2 子模块文档自检清单

### 产品需求场景

用户："帮我写一个登录功能的 PRD"

执行流程：
1. 识别需求 → 产品需求 → 引用 `prd-spec.md`
2. 检查触发条件（新增核心功能）→ 满足条件
3. 评估复杂度 → 中等复杂度 → 包含核心章节
4. 按 Part 4.1 结构写作：文档信息 → 背景与目标 → 需求详述 → 范围界定 → 非功能需求
5. 执行 Part 6 自检清单

### 测试规划场景

用户："为订单模块设计测试用例"

执行流程：
1. 识别需求 → 测试文档 → 引用 `test-spec.md`
2. 检查触发条件（新增核心模块）→ 满足条件
3. 判断文档类型 → 测试用例
4. 按 Part 4.3 格式编写测试用例表（ID/标题/前置条件/输入/步骤/预期结果/优先级）
5. 执行 Part 6.2 测试用例自检清单
