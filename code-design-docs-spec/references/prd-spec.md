# 产品需求文档规范

> 本规范适用于软件产品的需求定义与 PRD 撰写。
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

---

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **User Story（用户故事）** | 从用户视角描述功能的简短陈述，格式"作为 [角色]，我希望 [功能]，以便 [价值]" |
| **Acceptance Criteria（验收标准）** | 定义功能完成条件的可验证检查项列表 |
| **Epic（史诗）** | 大型功能集合，可拆分为多个 User Story |
| **Feature（特性）** | 可独立交付的用户价值单元 |
| **MVP（最小可行产品）** | 满足核心用户需求的最小功能集合 |
| **Persona（用户画像）** | 典型用户的虚构代表，包含特征、目标、痛点 |
| **Use Case（用例）** | 用户与系统交互完成特定目标的场景描述 |
| **PRD** | Product Requirements Document，产品需求文档 |
| **Success Criteria（成功标准）** | 衡量产品/功能是否成功的可量化指标 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

---

## Part 2：触发条件

PRD 的更新判断基于核心原则：**产品方向或需求范围变更时写文档，实现细节变更时不写**。

**REQUIRED** 新建或更新 PRD 的条件（满足任一即触发）：

- 新产品或新子系统从零开始规划
- 新增核心功能或重大功能变更
- 目标用户或用户需求发生变更
- 涉及 >= 3 个模块的跨功能需求
- 商业模式或盈利方式调整
- 竞品分析导致的产品策略调整
- 已有 PRD 中记录的需求被推翻或大幅调整

**PROHIBITED** 触发 PRD 更新的变更——

- 纯技术实现方案调整（不改变用户可见功能）
- UI 微调（如颜色、间距调整，不改变功能逻辑）
- 性能优化（不改变功能范围）
- Bug 修复
- 文案修改
- 单个字段校验规则调整

## Part 3：需求分析方法

### 3.1 用户故事格式

**REQUIRED** 所有功能需求使用标准用户故事格式描述：

```
作为 [用户角色]，
我希望 [功能描述]，
以便 [获得的价值/解决的问题]。
```

**示例**：

```
作为 注册新用户，
我希望 用手机号快速注册，
以便 无需记忆复杂密码即可使用应用。
```

**PROHIBITED** 使用以下非用户视角的描述：

- ✗ "系统应该支持手机号注册"
- ✗ "我们需要一个注册功能"
- ✗ "添加手机号注册模块"

### 3.2 验收标准定义

**REQUIRED** 每个 User Story 必须包含验收标准，使用 Given-When-Then 格式或检查项列表。

#### Given-When-Then 格式

```
Given [前置条件]
When [用户操作]
Then [预期结果]
```

**示例**：

```
Given 用户未注册
When 用户输入有效手机号并点击"获取验证码"
Then 用户收到短信验证码
And 验证码有效期为 5 分钟
```

#### 检查项列表格式

适用于简单场景：

```
- [ ] 手机号格式校验：11 位数字，以 1 开头
- [ ] 验证码发送频率限制：60 秒内只能发送一次
- [ ] 验证码失效时间：5 分钟
- [ ] 错误提示：手机号已被注册时显示"该手机号已注册，请直接登录"
```

**PROHIBITED** 验收标准中的模糊描述：

- ✗ "响应很快"
- ✗ "界面美观"
- ✗ "用户体验良好"

### 3.3 需求优先级划分

**REQUIRED** 所有需求使用统一优先级标注：

| 优先级 | 定义 | 发布策略 |
|--------|------|---------|
| **P0** | 阻塞性需求，没有则产品无法使用 | MVP 必须包含 |
| **P1** | 核心需求，显著影响用户价值 | 首发版本必须包含 |
| **P2** | 重要需求，影响用户满意度 | 可后续迭代补充 |
| **P3** | 增值需求，提升用户体验 | 长期规划，视资源决定 |

**优先级判断标准**：

- **P0**：目标用户无法完成核心任务
- **P1**：显著影响用户留存或转化
- **P2**：有替代方案或用户可接受当前方式
- **P3**：锦上添花，无直接业务影响

### 3.4 范围界定

**REQUIRED** 明确界定需求范围的包含与不包含：

#### In Scope（包含范围）

- 本 PRD 要实现的功能列表
- 支持的平台/设备
- 覆盖的用户场景
- 支持的用户类型

#### Out of Scope（明确排除）

- **REQUIRED** 明确列出不做的事情
- **REQUIRED** 说明为什么不做（资源限制、后续迭代、非目标场景）
- 避免范围蔓延和预期管理问题

**示例**：

```markdown
## Out of Scope

- 第三方登录（微信/QQ）：首期仅支持手机号注册，后续迭代补充
- 国际手机号：首期仅支持中国大陆手机号
- 企业版功能：属于 B 端产品规划，不在本 C 端 PRD 范围
```

### 3.5 成功标准定义

**REQUIRED** 每个 PRD 必须定义可量化的成功标准，用于衡量产品/功能是否达到预期目标。

**成功标准类型**：

| 类型 | 示例 |
|------|------|
| **用户行为指标** | 功能使用率达到 30%，次日留存提升 5% |
| **业务指标** | 转化率提升 10%，客单价增加 15% |
| **技术指标** | 页面加载时间 <= 2s，崩溃率 < 0.1% |
| **满意度指标** | NPS 提升 10 分，用户满意度 >= 4.5/5 |

**PROHIBITED** 模糊的成功标准：

- ✗ "提升用户体验"
- ✗ "让用户更满意"
- ✗ "优化系统性能"

## Part 4：文档规范

### 4.1 PRD 标准结构

PRD 采用统一结构，各章节按顺序编写。简单功能可精简，复杂产品可扩展，通过 OPTIONAL 章节灵活适应不同规模。

**REQUIRED 章节（按顺序）**：

#### § 文档信息

位于文档最顶部，提供元数据：

```markdown
| 字段 | 内容 |
|------|------|
| **名称** | 产品/功能名称 |
| **版本** | x.y.z |
| **作者** | 姓名 |
| **创建日期** | YYYY-MM-DD |
| **最后更新** | YYYY-MM-DD |
| **审批人** | 姓名 |
| **状态** | draft / review / approved / deprecated |
| **关联文档** | 链接到相关 PRD、架构文档等 |
```

#### § 背景与目标

**REQUIRED** 包含以下内容：

- **背景**：当前问题和机会，为什么要做这个产品/功能
- **目标**：本 PRD 要解决的核心问题（1-3 句话）
- **成功标准**：2-5 个可量化的衡量指标（Part 3.5）
- **用户价值**：目标用户能获得什么收益

**PROHIBITED**：过于宽泛的目标（如"打造行业领先产品"）。

#### § 用户画像（OPTIONAL）

当产品涉及新的目标用户类型时 **REQUIRED**，否则 **OPTIONAL**。

- 描述目标用户类型（1-3 个主要 Persona）
- 每个 Persona 包含：
  - 基本信息（年龄、职业、技术熟练度）
  - 核心目标
  - 当前痛点
  - 使用场景

#### § 需求详述

**REQUIRED** 按模块或功能分组描述，每组包含：

- **功能名称**
- **用户故事**（Part 3.1 格式）
- **验收标准**（Part 3.2 格式）
- **优先级**（P0/P1/P2/P3）
- **依赖关系**（依赖其他哪些功能）
- **UI/UX 说明**（OPTIONAL）：关键界面描述或原型链接

#### § 范围界定

**REQUIRED** 明确界定：

- **In Scope**：详细功能列表，本 PRD 要实现的全部内容
- **Out of Scope**：明确排除的内容及原因
- **未来规划**（OPTIONAL）：已知但本期不做的后续计划

#### § 非功能需求（OPTIONAL）

当涉及性能、安全、兼容性等非功能要求时 **REQUIRED**。

- 性能要求（响应时间、并发数、吞吐量）
- 兼容性要求（浏览器、设备、系统版本）
- 安全要求（数据加密、权限控制、合规要求）
- 可用性要求（SLA、故障恢复、降级策略）

**REQUIRED** 所有非功能需求必须有具体的量化指标。

#### § 数据需求（OPTIONAL）

当需要新增数据采集或报表时 **REQUIRED**。

- 需要采集的用户行为数据
- 需要新增的埋点事件
- 数据报表需求
- 数据隐私和合规要求

#### § 风险评估（OPTIONAL）

当项目存在显著风险时 **REQUIRED**。

- 识别的主要风险（技术、业务、资源、时间）
- 风险影响评估
- 应对策略和预案

#### § 时间计划（OPTIONAL）

当需要明确时间节点时 **REQUIRED**。

- 关键里程碑
- 期望发布日期
- 依赖关系和时间约束

#### § 附录（OPTIONAL）

- 竞品分析
- 用户调研摘要
- 原型/设计稿链接
- 相关文档链接

### 4.2 文档篇幅控制

PRD 篇幅应根据需求复杂度灵活调整：

- **简单功能**（单一模块、开发周期 < 2 周）：聚焦核心章节，省略大部分 OPTIONAL 章节，2-4 页
- **中等复杂度**（2-3 个模块、开发周期 2-6 周）：包含核心章节和关键 OPTIONAL 章节，4-8 页
- **复杂产品**（新系统或大版本、开发周期 > 6 周）：完整覆盖所有章节，允许 10+ 页

**PROHIBITED** 为凑篇幅而填充无关内容。

### 4.3 目录结构与命名

**REQUIRED** 目录结构：

```
docs/design/prd/
├── 00-overview.md               # PRD 总览文档（唯一，导航入口）
├── 10-{product}.md              # 产品总体 PRD（两位数字序号，表示阅读顺序）
├── 20-{feature}.md              # 功能模块 PRD（两位数字序号）
├── 20.10-{sub-feature}.md       # 子功能 PRD（NN.MM 格式）
└── reference/                   # 参考文档
    └── ...
```

**命名规则**：

- 文件名全小写，单词间用连字符 `-` 分隔
- 数字编号仅表示阅读顺序，不表示版本
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-`、`-v{version}` 等版本标记——版本由 Git 管理
- **REQUIRED** 同一主题只允许一个 `active` 文档存在
- 废弃文档标记为 `deprecated` 状态，30 天以上无替代文档则直接删除

### 4.4 文档间关联

**REQUIRED** `README.md` 作为 PRD 的导航入口，列出所有 `active` 文档的链接与一句话说明。`deprecated` 文档不出现在导航中。

**交叉引用规则**：

- **REQUIRED** PRD 引用关联的架构设计文档
- **REQUIRED** 功能模块 PRD 引用主产品 PRD
- **REQUIRED** PRD 引用依赖的其他 PRD
- **PROHIBITED** 在 PRD 中复制架构设计细节
- **PROHIBITED** 在 PRD 中复制其他 PRD 的内容

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个 PRD 头部包含状态标注：

```markdown
> **Status**: `draft` | `active` | `deprecated`
> **Superseded by**: [链接]（仅 deprecated 文档填写）
```

**状态规则**：

- `draft → active`：自检清单（Part 6）全部通过后标记为 `active`
- `active → deprecated`：**REQUIRED** 指向替代文档；30 天以上无替代文档则直接删除
- 废弃文档不出现在 README.md 导航中

### 5.2 语言精确性

**PROHIBITED** 在需求描述中使用模糊词汇：大概、可能、应该、通常、视情况而定、尽量、适当。

**REQUIRED** 需求描述使用确定性语言，可明确判断是否满足。

| ✗ | ✓ |
|---|---|
| "页面加载应该很快" | "首屏加载时间 <= 2s（3G 网络）" |
| "支持大多数浏览器" | "支持 Chrome 90+, Safari 14+, Edge 90+" |
| "用户可以方便地操作" | "核心功能可在 3 步内完成" |
| "数据可能需要备份" | "关键数据每日自动备份，保留 30 天" |

### 5.3 用户故事格式

**REQUIRED** 用户故事必须完整包含三个要素：

```markdown
作为 [具体用户角色]，
我希望 [具体可执行的功能]，
以便 [明确的价值/解决的问题]。
```

**质量检查**：

- 角色是否具体？（"用户"太泛，"首次注册的新用户"更具体）
- 功能是否可测试？（"更好的体验"不可测，"一键下单"可测）
- 价值是否明确？（"方便"不明确，"节省 50% 操作时间"明确）

### 5.4 验收标准格式

**REQUIRED** 每个验收标准必须是：

1. **可验证**：能通过测试或观察判断是否满足
2. **无歧义**：不同人理解一致
3. **可独立验证**：不依赖其他验收标准的执行结果

**数量建议**：

- 简单功能：3-5 条验收标准
- 复杂功能：5-10 条验收标准
- **PROHIBITED** 超过 15 条（考虑拆分功能）

### 5.5 内容边界

下表定义哪些内容属于 PRD，哪些属于其他文档：

| 内容类型 | PRD | 其他文档 |
|---------|-----|---------|
| 用户需求描述 | REQUIRED | PROHIBITED |
| 用户故事和验收标准 | REQUIRED | PROHIBITED |
| 功能优先级 | REQUIRED | PROHIBITED |
| 用户画像 | REQUIRED | 可选 |
| 成功标准定义 | REQUIRED | PROHIBITED |
| 技术实现方案 | PROHIBITED | REQUIRED（架构文档） |
| 数据库设计 | PROHIBITED | REQUIRED（架构文档） |
| API 接口定义 | PROHIBITED | REQUIRED（API 文档） |
| UI 设计稿 | 链接/引用 | REQUIRED（设计系统） |
| 测试用例 | PROHIBITED | REQUIRED（测试文档） |

## Part 6：自检清单

Agent 在生成或审查 PRD 后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 内容完整性

- [ ] 包含文档信息表（名称、版本、作者、状态）？
- [ ] 背景说明清晰，说明了为什么要做这个产品/功能？
- [ ] 目标定义明确且聚焦（1-3 句话）？
- [ ] 成功标准可量化（有具体数值指标）？
- [ ] 每个功能都有用户故事（三段式格式）？
- [ ] 每个用户故事都有验收标准？
- [ ] 所有需求标注了优先级（P0/P1/P2/P3）？
- [ ] Out of Scope 明确列出了排除的内容及原因？

### 6.2 质量检查

- [ ] Status 标注存在且有效？
- [ ] 无模糊词汇（大概、可能、应该、通常等）？
- [ ] 用户故事包含三个要素（角色、功能、价值）？
- [ ] 验收标准可验证且无歧义？
- [ ] 无技术实现细节（数据库、API、代码）？
- [ ] 非功能需求有具体的量化指标（如有）？
- [ ] 文档间引用使用相对链接？
- [ ] README.md 导航与文档状态一致？

### 6.3 风险评估（如有）

- [ ] 识别了主要风险？
- [ ] 风险影响评估合理？
- [ ] 有明确的应对策略？

## Agent 执行工作流

Agent 处理 PRD 任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新 PRD，结束
   └── 满足触发条件 → 继续

2. 评估复杂度，确定章节范围（Part 4.2）
   ├── 简单功能 → 聚焦核心章节，省略部分 OPTIONAL
   ├── 中等复杂度 → 包含核心章节和关键 OPTIONAL
   └── 复杂产品 → 完整覆盖所有章节

3. 按 PRD 标准结构写作（Part 4.1）
   ├── 使用标准用户故事格式（Part 3.1）
   ├── 编写可验证的验收标准（Part 3.2）
   ├── 标注明确的优先级（Part 3.3）
   ├── 界定清晰的范围边界（Part 3.4）
   ├── 定义量化的成功标准（Part 3.5）
   └── 遵守语言精确性规则（Part 5.2）

4. 执行自检清单（Part 6）
   ├── 所有项通过 → 输出 PRD
   └── 存在未通过项 → 修正后重新检查
```
