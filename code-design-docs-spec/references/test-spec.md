# 测试文档规范

> 本规范适用于软件系统的测试策略设计与测试文档撰写。
> 规范同时作为 Agent 可执行的规则集使用——每条规则均标注 **REQUIRED** / **PROHIBITED** / **OPTIONAL**，Agent 可机械地对照执行与自检。

---

## Part 1：术语表

本规范中所有术语以此定义为准，Agent 和人均使用同一词汇：

| 术语 | 定义 |
|------|------|
| **Test Case（测试用例）** | 验证特定功能或行为的输入、执行步骤和预期结果的集合 |
| **Test Suite（测试套件）** | 针对同一模块或功能的测试用例集合 |
| **Fixture（测试固件）** | 测试执行前需要准备的预置数据和环境状态 |
| **Mock** | 模拟外部依赖的测试替身，用于验证交互行为 |
| **Stub** | 返回固定值的测试替身，用于隔离被测单元 |
| **Unit Test（单元测试）** | 测试最小可测单元（通常是函数或方法） |
| **Integration Test（集成测试）** | 测试多个模块或服务之间的交互 |
| **E2E Test（端到端测试）** | 模拟真实用户场景的完整流程测试 |
| **Coverage（覆盖率）** | 被测试执行到的代码占总代码的比例 |

**规则强度**：**REQUIRED**（必须）/ **PROHIBITED**（禁止）/ **OPTIONAL**（可选）。无标记的条目为补充说明。

---

## Part 2：触发条件

测试文档的更新判断基于核心原则：**测试策略变更或测试范围扩展时写文档，测试实现细节变更时不写**。

**REQUIRED** 新建或更新测试文档的条件（满足任一即触发）：

- 新系统或新子系统从零开始设计
- 新增核心模块或对外暴露接口的模块
- 测试策略发生变更（如引入新的测试层级、覆盖率目标调整）
- 单次变更涉及 >= 3 个模块的集成测试场景
- 安全相关功能的测试策略制定
- 性能测试或压力测试策略制定

**PROHIBITED** 触发测试文档更新的变更——

- 单个测试用例的新增、修改或删除（除非测试策略改变）
- 测试实现细节的优化
- Bug 修复对应的测试调整
- 测试数据的微调

## Part 3：测试策略与方法

### 3.1 测试金字塔

**REQUIRED** 测试策略遵循测试金字塔原则，各层比例建议：

- **单元测试（70%）**：最底层，数量最多，执行最快
- **集成测试（20%）**：中间层，验证模块交互
- **E2E 测试（10%）**：顶层，覆盖关键用户流程

**PROHIBITED** 颠倒金字塔（大量 E2E 测试、少量单元测试）。

### 3.2 测试范围界定

**REQUIRED** 每个测试层级明确定义测试范围边界：

#### 单元测试范围

- **REQUIRED** 覆盖所有公共函数和方法
- **REQUIRED** 覆盖边界条件和异常路径
- **PROHIBITED** 测试私有实现细节
- **PROHIBITED** 测试简单的 getter/setter（除非包含业务逻辑）

#### 集成测试范围

- **REQUIRED** 覆盖模块间接口契约
- **REQUIRED** 覆盖数据流转换
- **PROHIBITED** 测试已被单元测试覆盖的内部逻辑

#### E2E 测试范围

- **REQUIRED** 仅覆盖关键用户流程（Happy Path）
- **OPTIONAL** 覆盖高频异常场景
- **PROHIBITED** 覆盖所有边界条件（应在单元测试层覆盖）

### 3.3 测试数据管理

**REQUIRED** 测试文档说明测试数据来源：

- **Fixture**：预定义的静态数据，用于可重复测试
- **Factory**：动态生成的数据，支持参数化测试
- **Snapshot**：基于上次运行结果的对比测试

**PROHIBITED** 使用生产环境数据作为测试数据。

### 3.4 覆盖率策略

**REQUIRED** 定义覆盖率目标（按层级）：

| 层级 | 最低覆盖率 | 目标覆盖率 |
|------|-----------|-----------|
| 单元测试 | 70% | 85% |
| 集成测试 | 60% | 75% |
| 整体 | 75% | 90% |

**PROHIBITED** 以覆盖率为唯一质量指标。高覆盖率不保证测试质量。

## Part 4：文档规范

### 4.1 文档分类

测试文档分为三类，按不同目的撰写：

| 文档 | 回答的问题 | 存放位置 |
|------|-----------|---------|
| **测试计划** | "测什么、怎么测、用什么资源" | `docs/design/test/plan-*.md` |
| **测试用例** | "具体输入是什么、预期输出是什么" | `docs/design/test/cases-*.md` |
| **测试报告** | "测了什么、结果如何、质量如何" | `docs/design/test/report-*.md` |

### 4.2 测试计划（Test Plan）

**定位**：描述测试策略、范围和资源的总体文档。

**REQUIRED 章节（按顺序）**：

#### § 测试目标

- 一句话描述测试目的
- 列出测试范围（包含/不包含的功能）
- **PROHIBITED**：过于宽泛的目标（如"确保系统稳定"）

#### § 测试策略

- 测试金字塔各层级的比例和覆盖范围
- 各层级的测试类型和工具选择
- **REQUIRED** 说明为什么选择这种策略

#### § 测试范围

- **包含范围**：明确列出要测试的功能模块
- **排除范围**：明确列出不测试的内容及原因
- **PROHIBITED**：模糊的边界描述（如"主要功能"）

#### § 测试资源

- 测试环境要求
- 测试数据需求
- 自动化测试工具

#### § 准入/准出标准

- **准入标准**：开始测试前必须满足的条件
- **准出标准**：测试完成、可发布的条件
- **REQUIRED** 所有标准必须可量化验证

#### § 风险评估（OPTIONAL）

- 识别测试风险
- 制定风险应对策略

### 4.3 测试用例（Test Cases）

**定位**：具体可执行的测试场景描述。

**REQUIRED 格式**：

每个测试用例使用统一表格格式：

| 字段 | 说明 | 必填 |
|------|------|------|
| **ID** | 唯一标识符，格式 `TC-{模块}-{序号}` | REQUIRED |
| **标题** | 一句话描述测试目的 | REQUIRED |
| **前置条件** | 执行测试前必须满足的条件 | REQUIRED |
| **输入数据** | 具体的输入值或参数 | REQUIRED |
| **执行步骤** | 详细的操作步骤（<= 10 步） | REQUIRED |
| **预期结果** | 明确的可验证的预期输出 | REQUIRED |
| **优先级** | P0(阻塞)/P1(高)/P2(中)/P3(低) | REQUIRED |
| **关联需求** | 关联的需求或用户故事 ID | OPTIONAL |

**优先级定义**：

- **P0**：阻塞性测试，失败则阻止发布
- **P1**：核心功能测试，失败则严重影响用户体验
- **P2**：常规功能测试
- **P3**：边缘场景测试

### 4.4 测试报告（Test Report）

**定位**：测试执行后的结果汇总。

**REQUIRED 章节**：

#### § 执行摘要

- 测试时间范围
- 执行的用例总数
- 通过率统计
- 主要问题概述

#### § 详细结果

- 按模块或功能的测试结果
- 失败的用例列表及失败原因
- 阻塞的问题及影响

#### § 质量评估

- 是否满足准出标准
- 遗留风险说明
- 发布建议

#### § 改进建议（OPTIONAL）

- 测试流程改进建议
- 未覆盖场景的补充计划

### 4.5 目录结构与命名

**REQUIRED** 目录结构：

```
docs/design/test/
├── 00-overview.md             # 测试文档总览（唯一，导航入口）
├── 10-plan-{system}.md        # 测试计划（两位数字序号）
├── 20-cases-{module}.md       # 测试用例（两位数字序号，按模块拆分）
├── 30-report-{version}.md     # 测试报告（两位数字序号，按版本或日期）
└── reference/                 # 参考文档
    └── ...
```

**命名规则**：

- 文件名全小写，单词间用连字符 `-` 分隔
- 数字编号仅表示阅读顺序，不表示版本
- **PROHIBITED** 使用 `v1/`、`v2/`、`old-` 等版本前缀目录——版本由 Git 管理
- **REQUIRED** 同一主题只允许一个 `active` 文档存在
- 废弃文档标记为 `deprecated` 状态，30 天以上无替代文档则直接删除

### 4.6 文档间关联

**REQUIRED** `README.md` 作为测试文档的导航入口，列出所有文档链接与一句话说明。

**交叉引用规则**：

- **REQUIRED** 测试用例引用关联的需求文档
- **REQUIRED** 测试报告引用执行的测试计划
- **PROHIBITED** 复制大段内容，使用相对链接指向原文

## Part 5：写作规则

### 5.1 文档状态标注

**REQUIRED** 每个文档头部包含状态标注：

```markdown
> **Status**: `draft` | `active` | `deprecated`
> **Superseded by**: [链接]（仅 deprecated 文档填写）
```

**状态规则**：

- `draft → active`：自检清单（Part 6）全部通过后标记为 `active`
- `active → deprecated`：**REQUIRED** 指向替代文档；30 天以上无替代文档则直接删除
- 废弃文档不出现在 README.md 导航中

### 5.2 语言精确性

**PROHIBITED** 在测试用例中使用模糊词汇：大概、可能、应该、通常、视情况而定。

**REQUIRED** 测试用例使用确定性语言：

| ✗ | ✓ |
|---|---|
| "页面应该显示正确" | "页面显示'操作成功'提示" |
| "数据可能被保存" | "数据库中新增一条记录，status='active'" |
| "响应时间通常很快" | "响应时间 <= 200ms" |

### 5.3 测试用例格式

**标题格式**：

- **REQUIRED** 使用 "[动作] + [预期结果]" 格式
- **示例**：`验证用户登录成功后跳转到首页`
- **PROHIBITED**：`测试登录功能`

**执行步骤格式**：

- **REQUIRED** 每条步骤以动词开头
- **REQUIRED** 步骤编号连续
- **PROHIBITED** 一条步骤包含多个操作

```markdown
✓
1. 打开登录页面
2. 输入有效的用户名
3. 输入有效的密码
4. 点击"登录"按钮

✗
1. 打开登录页面并输入用户名密码然后点击登录
```

**预期结果格式**：

- **REQUIRED** 每个预期结果可验证（能明确判断是否满足）
- **REQUIRED** 包含具体的数据值或状态
- **PROHIBITED** 主观描述（如"界面美观"）

### 5.4 内容边界

下表定义哪些内容属于测试文档，哪些属于实现文档：

| 内容类型 | 测试文档 | 实现文档 |
|---------|---------|---------|
| 测试策略和范围 | REQUIRED | 可选 |
| 测试用例描述 | REQUIRED | PROHIBITED |
| 测试数据定义 | REQUIRED | 可选 |
| 测试执行结果 | REQUIRED | PROHIBITED |
| 测试代码实现 | PROHIBITED | REQUIRED |
| 测试工具配置 | PROHIBITED | REQUIRED |
| Mock/Stub 实现 | PROHIBITED | REQUIRED |

## Part 6：自检清单

Agent 在生成或审查测试文档后，**REQUIRED** 对照以下清单逐项检查。每项答案为是/否，否则不通过。存在未通过项时，**REQUIRED** 修正对应内容后重新检查。

### 6.1 测试计划

- [ ] 包含明确的测试目标和范围（包含/排除）？
- [ ] 测试策略符合金字塔原则（70/20/10）？
- [ ] 准入/准出标准可量化验证？
- [ ] 风险评估识别了主要风险？
- [ ] 无模糊的边界描述？

### 6.2 测试用例

- [ ] 每个用例有唯一 ID？
- [ ] 标题使用"[动作] + [预期结果]"格式？
- [ ] 执行步骤 <= 10 步且每步单一操作？
- [ ] 预期结果可验证且包含具体数据？
- [ ] 优先级标注为 P0/P1/P2/P3？
- [ ] 无模糊词汇（大概、可能、应该等）？

### 6.3 测试报告

- [ ] 包含执行摘要（时间、数量、通过率）？
- [ ] 失败用例有明确的失败原因？
- [ ] 质量评估基于准出标准？
- [ ] 发布建议清晰明确？

### 6.4 通用质量

- [ ] Status 标注存在且有效？
- [ ] 文档间引用使用相对链接？
- [ ] README.md 导航完整？
- [ ] 无实现细节（测试代码、工具配置）？

## Agent 执行工作流

Agent 处理测试文档任务时，遵循以下工作流：

```
1. 判断触发条件（Part 2）
   ├── 不满足触发条件 → 不需要写/更新文档，结束
   └── 满足触发条件 → 继续

2. 判断文档类型（Part 4.1）
   ├── 测试策略/范围 → 测试计划（Part 4.2）
   ├── 具体测试场景 → 测试用例（Part 4.3）
   └── 测试结果汇总 → 测试报告（Part 4.4）

3. 按对应章节结构写作
   ├── 遵守语言精确性规则（Part 5.2）
   ├── 使用正确的测试用例格式（Part 5.3）
   └── 遵守内容边界规则（Part 5.4）

4. 执行自检清单（Part 6）
   ├── 所有项通过 → 输出文档
   └── 存在未通过项 → 修正后重新检查
```
